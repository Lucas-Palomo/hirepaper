# 010 - Add PDF generation metadata for ats-check

## Status
Completed

## Context
The current `ats-check` command validates the generated PDF as a standalone artifact, which is correct from a CLI boundary perspective.

However, two gaps remain:
- `ats-check` does not know whether the PDF was actually produced by `curriculum-gen`;
- keyword validation is currently hardcoded instead of being based on the data and rendering context used during PDF generation.

Because `ats-check` is a separate command, it should not depend on access to the original candidate JSON file at validation time.

The correct place to bridge generation and validation is the PDF metadata.

## Goal
Embed generation metadata in the produced PDF so `ats-check` can validate the artifact based on information stored inside the PDF itself.

## Core Direction
The generated PDF should carry metadata that allows `ats-check` to answer at least these questions:
- was this PDF generated by `curriculum-gen`?
- which engine was used?
- which keywords were expected to remain visible in the extracted text?

Validation should use PDF metadata, not the original input JSON.

## Required Metadata
### 1. Generator identity
The PDF must include metadata that identifies it as generated by this project.

This should allow `ats-check` to fail or warn clearly when the file is not recognized as a `curriculum-gen` artifact.

Required field:
- `X App`

Expected form in metadata inspection:

```text
X App: curriculum-gen
```

Additional standard metadata such as producer or creator may still be written, but `ats-check` should treat `X App` as the explicit identity hint for this phase.

Important:
- if `X App` is absent, `ats-check` must still continue with generic PDF validation;
- if `X App` is different from `curriculum-gen`, `ats-check` must still continue with generic PDF validation;
- only the strict project-specific metadata contract should depend on `X App: curriculum-gen`.
- when `X App` is exactly `curriculum-gen`, `ats-check` must activate the project-specific metadata verification rules described below.

### 2. Engine metadata
The PDF must store the LaTeX engine used during generation.

Requested naming convention:
- `X Engine`

This field should contain a clear value such as:
- `xelatex`
- `pdflatex`

Expected form in metadata inspection:

```text
X Engine: xelatex
```

`ats-check` should read and report this value.

### 3. Keyword metadata by source
The PDF must store keyword metadata grouped by origin.

Required fields:
- `X Keywords Skills`
- `X Keywords Experience`
- `X Keywords Projects`

Expected form in metadata inspection:

```text
X Keywords Skills: Python, JavaScript, PostgreSQL
X Keywords Experience: Kafka, Docker, Kubernetes, AWS
X Keywords Projects: FastAPI, SQLAlchemy, Next.js
```

These fields should be generated from the actual rendering context, not from a fixed hardcoded list.

### 4. Standard header usage
In addition to the custom metadata field above, the keywords should also be stored in an appropriate standard PDF metadata header when practical.

For now, this task should include:
- custom metadata fields:
  - `X Keywords Skills`
  - `X Keywords Experience`
  - `X Keywords Projects`
- standard metadata field: `Keywords`

For this phase, `Keywords` must contain the canonical aggregated keyword list derived from the source-specific metadata above.

Expected form in metadata inspection:

```text
Keywords: Python, Kafka, PostgreSQL, Docker
```

Important:
- `Keywords` is the canonical aggregated keyword field for this phase;
- `X Keywords Skills`, `X Keywords Experience`, and `X Keywords Projects` are diagnostic source-specific fields;
- do not introduce a separate aggregated `X Keywords` field in this phase.

## Tooling Requirement
Use `exiftool` to inspect and validate PDF metadata.

`ats-check` should use `exiftool` output as the source of truth for metadata inspection.

Example validation direction:

```bash
exiftool output/resume.pdf
```

The command should read:
- `X App`;
- `X Engine`;
- `X Keywords Skills`;
- `X Keywords Experience`;
- `X Keywords Projects`;
- `Keywords`.

## ats-check Changes
The `ats-check` command should be extended to:

1. verify that the PDF appears to be generated by `curriculum-gen`;
2. read the engine from metadata;
3. read the expected keywords from metadata;
4. use the metadata-provided keyword list instead of a hardcoded internal list;
5. use `Keywords` as the primary aggregated keyword source;
6. use the `X Keywords *` fields as diagnostic source-specific context;
7. report missing metadata clearly;
8. degrade gracefully when metadata is absent, but make that absence visible as a warning or failure according to the chosen policy.

When `X App` is exactly `curriculum-gen`, `ats-check` should additionally:

1. verify the presence of `X Keywords Skills`;
2. verify the presence of `X Keywords Experience`;
3. verify the presence of `X Keywords Projects`;
4. compare `Keywords` against the deterministic union of those three source-specific fields.

## Metadata Source Rules
Keyword metadata should be derived during generation from the actual candidate/rendering context.

For this phase, use this narrower and deterministic derivation rule:
- `X Keywords Skills` from `skills.categories[].items`;
- `X Keywords Experience` from `experience[].technologies`;
- `X Keywords Projects` from `projects[].technologies`;
- `Keywords` derived deterministically from the union of those three sources.

The task does not need to solve perfect keyword ranking.

It does need to ensure:
- the metadata is candidate-specific;
- the metadata is generation-specific;
- `ats-check` validates against what the PDF was expected to contain.

## PDF Boundary
The metadata must travel with the PDF itself.

That is the point of this task:
- `generate` writes the metadata;
- `ats-check` reads only the PDF and validates from that artifact.

Do not redesign `ats-check` to require the input JSON as an argument.

## Failure and Warning Policy
Suggested behavior:

- if `X App` is missing: `WARN`, but continue generic validation;
- if `X App` exists but is not `curriculum-gen`: `WARN`, but continue generic validation;
- if `X Engine` is missing: `WARN`;
- if `X App` is `curriculum-gen` and any `X Keywords *` field is missing: `WARN`;
- if `X App` is `curriculum-gen` and any `X Keywords *` field exists but is empty: `WARN`;
- if `X App` is `curriculum-gen` and `Keywords` is missing: `WARN`;
- if `X App` is `curriculum-gen` and `Keywords` differs from the deterministic union of `X Keywords Skills`, `X Keywords Experience`, and `X Keywords Projects`: `WARN`;
- if metadata keywords exist but are not found in extracted text: `WARN` or `FAIL` depending on severity policy.

For this phase, use these specific rules:
- `X App` absent: `WARN`
- `X App` different from `curriculum-gen`: `WARN`
- `X Engine` absent: `WARN`
- `X App=curriculum-gen` and missing `X Keywords Skills`: `WARN`
- `X App=curriculum-gen` and missing `X Keywords Experience`: `WARN`
- `X App=curriculum-gen` and missing `X Keywords Projects`: `WARN`
- `X App=curriculum-gen` and any present `X Keywords *` field is empty: `WARN`
- `X App=curriculum-gen` and missing `Keywords`: `WARN`
- `X App=curriculum-gen` and `Keywords` differs from the deterministic union of all `X Keywords *`: `WARN`

## doctor Changes
Because this task depends on `exiftool`, `doctor` should also check whether `exiftool` is available.

If missing, `doctor` should report how to install it.

## Constraints
- Keep `ats-check` independent from the original JSON input.
- Do not keep hardcoded ATS keyword lists as the primary validation source.
- Use metadata as the contract between generation and validation.
- Prefer clear, inspectable metadata over opaque encoding tricks.
- Keep the implementation local and deterministic.

## Acceptance Criteria
The task should be considered complete only if:

1. Generated PDFs include metadata identifying `curriculum-gen`.
1. Generated PDFs include `X App: curriculum-gen`.
2. Generated PDFs include `X Engine`.
3. Generated PDFs include `X Keywords Skills`.
4. Generated PDFs include `X Keywords Experience`.
5. Generated PDFs include `X Keywords Projects`.
6. Generated PDFs include `Keywords`.
7. `Keywords` is derived deterministically from the source-specific fields.
8. `ats-check` reads metadata using `exiftool`.
9. `ats-check` continues generic validation even when `X App` is absent or different.
10. `ats-check` applies project-specific metadata checks when `X App` is `curriculum-gen`.
11. `ats-check` uses metadata-provided keywords instead of the current hardcoded list.
12. `doctor` checks for `exiftool`.
13. When `X App=curriculum-gen`, missing or empty `X Keywords *` fields produce `WARN`.
14. When `X App=curriculum-gen`, missing `Keywords` produces `WARN`.
15. When `X App=curriculum-gen`, `Keywords` differing from the union of all `X Keywords *` produces `WARN`.

## Suggested Verification
The implementing agent should verify:

```bash
curriculum-gen generate data/candidate.json -o output/resume.pdf -l pt-BR --density compact
exiftool output/resume.pdf
curriculum-gen ats-check output/resume.pdf
curriculum-gen doctor
```

The `exiftool` output should visibly show:
- `X App`;
- engine metadata;
- `X Keywords Skills`;
- `X Keywords Experience`;
- `X Keywords Projects`;
- `Keywords`.

## Notes For The Implementing Agent
- `ats-check` is a standalone artifact validator. Keep it that way.
- The PDF metadata is the handoff contract between generation and validation.
- `X App` is an identity hint for project-specific validation, not a reason to skip generic PDF checks.
- `X Engine` is required custom metadata for this phase.
- `Keywords` is the canonical aggregated keyword field.
- `X Keywords Skills`, `X Keywords Experience`, and `X Keywords Projects` are source-specific diagnostic fields.
- Do not add a separate aggregated `X Keywords` field in this phase.
