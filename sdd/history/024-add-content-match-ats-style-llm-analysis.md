# 024 - Add content match ATS-style LLM compatibility analysis

**Date:** 2026-06-05
**Agent:** Codex GPT-5.4 + opencode (deepseek-v4-flash)

---

## Context

The `content match` command was expanded from a connectivity test into an
ATS-style compatibility analysis workflow. During review, two contract gaps were
found in the initial delivery:

- `--inference` accepted unsupported values instead of failing clearly.
- the public versioned JSON Schema existed in the repository, but it was not
  actually used as the shared source of truth for prompt context and local
  validation.

This update closes those gaps and aligns the implementation with the backlog
task requirements. It also refines two operational details after follow-up
review:

- `timeout_seconds` and `max_tokens` are no longer hardcoded in the matching
  flow.
- `disclaimer` is no longer generated by the LLM and is now localized in code.

## Changes

### Modified: `src/curriculum_gen/content_match.py`

- Added strict validation for `--inference` so only `low|medium|high` are
  accepted.
- Added `load_result_schema()` to load and validate the public schema from
  `assets/schemas/content-match-result.schema.json`.
- Added `build_llm_result_schema()` so the model is asked for a leaner response
  contract without `disclaimer`, while the public output contract still keeps
  the localized disclaimer field.
- Updated `build_messages()` to include the exact versioned JSON Schema in the
  LLM request context, making the structural contract explicit to the model.
- Replaced the manual structural validator with schema-driven validation
  against the same public schema used in the prompt context.
- Added localized disclaimer generation in code, so the final public result is
  deterministic and does not spend tokens on a static field.
- Replaced hardcoded timeout/token overrides with effective runtime settings
  derived from `config.json` and optional CLI overrides.
- Reintroduced the previous operational fallback values for `content match`
  and `content tailor` only when `config.json` does not define
  `timeout_seconds` and/or `max_tokens`.
- Kept the score-band consistency check as an additional semantic validation
  layer after schema validation.
- Added `result-schema.json` to the ZIP log archive so the execution record
  includes the exact contract used for the run.
- Fixed `meta.json` lint reporting so `lint_status` distinguishes between
  `pass` and `warning` instead of marking every successful lint run as
  `warning`.

### Modified: `src/curriculum_gen/cli.py`

- Added `--timeout-seconds` to override the request timeout for a single run.
- Added `--max-tokens` to override the response token limit for a single run.
- Kept precedence as: CLI flag override, then `config.json`, then existing
  config-loader defaults.
- `content tailor` now also applies command-specific fallback values when the
  config omits timeout/token settings, matching the operational behavior of
  `content match`.

### Modified: `assets/prompts/content-match-default.txt`

- Removed `disclaimer` from the LLM output contract because it is now generated
  locally after model validation.

## Decisions and Tradeoffs

- **Schema as source of truth:** local validation now depends on the repository
  schema asset instead of a parallel hand-written validator. This removes drift
  between the documented contract and runtime behavior.
- **Public result vs model result:** the model now returns a reduced schema
  without `disclaimer`, and the final public result is assembled locally with a
  localized disclaimer before final validation/output. This reduces token usage
  and removes output variability for a static field.
- **Config-first runtime limits:** timeout and token ceilings now default to the
  values from `config.json`, with per-run CLI overrides when needed. When the
  config omits those fields, `content match` and `content tailor` fall back to
  the previous operational values (`300` seconds and `16384` tokens) instead of
  the lighter global loader defaults. This keeps operational policy under user
  control without regressing the original behavior of the content commands.
- **No new runtime dependency:** the validator walks the repository schema
  directly instead of introducing an external JSON Schema package. That keeps
  the command executable in the current environment while still using the
  versioned schema as the validation source of truth.
- **Keep semantic checks outside schema:** JSON Schema enforces structure and
  field-level constraints, but score-to-rating consistency is still enforced in
  Python because it is simpler and clearer than encoding band logic indirectly
  in the schema.

Expected outcomes:

- invalid `--inference` raises a clear `ContentMatchError`
- extra JSON properties are rejected
- empty strings that violate schema `minLength` are rejected
- disclaimer text is generated locally per locale instead of coming from the model

## Residual Risks

- There is still no automated test suite covering the `content match` workflow.
- The command depends on the model returning JSON in a single response; prompt
  or model behavior changes may still surface truncation or formatting errors.
- `content tailor` remains a connectivity test and is still outside the scope of
  task 024.
